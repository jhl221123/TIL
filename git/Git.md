# 버전 관리 시스템(VCS)
파일 변화를 시간에 따라 기록했다가 나중에 특정 시점의 버전을 다시 꺼내올 수 있는 시스템이다. VCS를 사용하면 프로젝트나 파일을 이전 상태로 되돌릴 수 있다. 또 시간에 따른 수정 내용을 비교해 볼 수 있어, 문제가 어디서 발생했는지 파악할 수 있다.

### Git은 분산 버전 관리 시스템(DVCS)을 사용한다.

![Alt text](<이미지/DVCS.png>)

클라이언트가 저장소를 히스토리와 더불어 전부 복제하기 때문에 서버에 문제가 생기더라도 이 복제물로 다시 작업을 시작할 수 있다. 대부분의 DVCS 환경에는 원격 저장소가 존재하며 동시에 다양한 그룹과 다양한 방법으로 협업할 수 있다.

# Git의 핵심
Git의 핵심은 Git을 이해하는데 굉장히 중요하다. 

다른 VCS와 Git의 가장 큰 차이점은 데이터를 다루는 방법에 있다. 다른 VCS는 **각 파일의 변화**를 시간순으로 관리하면서 파일들의 집합을 관리하는데 이를 델타 방식이라고 한다.

![Alt text](<이미지/다른 VCS의 관리 방식.png>)

반면 Git은 스냅샷 방식을 사용한다.

데이터를 스냅샷의 연속으로 취급하며, **파일이 존재하는 그 순간**을 중요하게 여긴다. 변경 사항이 없다면 Git은 성능을 위해서 파일을 새로 저장하지 않고, 이전 파일에 대한 링크만 저장한다.

![Alt text](<이미지/Git의 관리 방식.png>)

변경 사항만 관리하는 델타 방식이 효율적으로 보일 수 있지만 특정 버전으로 이동해야 할 경우 해당 버전까지의 모든 변경 사항을 반영해서 결과를 도출해야한다. 하지만 스냅샷 방식은 해당 버전을 불러오기만 하면 된다.

# Git의 무결성
Git은 데이터를 저장하기 전에 항상 체크섬을 구하고 그 체크섬으로 데이터를 관리한다. Git 없이는 체크섬을 다룰 수 없기 때문에 Git 없이는 어떠한 파일이나 디렉토리도 변경할 수 없다.

![Alt text](<이미지/체크섬.png>)

> 체크섬이란?<br>
중복 검사의 한 형태로 자료의 무결성을 보호하는 방법이다. Git의 커밋 아이디가 이에 해당하며, SHA-1 해시를 사용해서 40자 길이의 16진수 문자열로 표현한다.

# 파일의 세 가지 상태
Git은 파일을 세 가지 상태로 관리한다.

* Committed: 데이터가 로컬 데이터베이스에 안전하게 저장됐다는 것을 의미한다.

* Modified: 수정한 파일을 아직 로컬 데이터베이스에 커밋하지 않은 것을 말한다.

* Staged: 현재 수정한 파일을 곧 커밋할 것이라고 표시한 상태를 의미한다.

이 세 가지 상태는 Git 프로젝트의 다음 세 가지 단계와 연결되어 있다.

#### Git 디렉토리
Git이 프로젝트의 메타데이터와 객체 데이터베이스를 저장하는 곳으로 Git의 핵심이다. Git 디렉토리에 있는 파일들은 Committed 상태이다.

#### 워킹 트리
프로젝트의 특정 버전을 Checkout 한 것이다. 아직 Staging Area에 추가하지 않았으면 Modified 상태이다.

#### Staging Area
Git 디렉토리에 있으며 곧 커밋할 파일에 대한 정보를 저장한다. 파일을 수정하고 Staging Area에 추가했다면 Staged 상태이다.

    Git으로 워킹 트리에서 파일을 수정하고 Staging Area에 스냅샷을 만든다.
    Staging Area에 있는 파일들을 커밋해서 Git 디렉토리에 영구적인 스냅샷으로 저장한다.
